# Gis Catalog API

REST API для справочника организаций, зданий и видов деятельности с поддержкой геопоиска.

## О проекте

Проект реализует справочник организаций с возможностью поиска по различным критериям, включая географический поиск на карте.

### Реализованный функционал

- Получение полной информации об организации по ID
- Поиск организаций по названию
- Поиск организаций в конкретном здании
- Поиск организаций по виду деятельности (с учётом иерархии)
- Геопоиск организаций в радиусе от точки
- Геопоиск организаций в прямоугольной области (bounding box)
- Иерархия видов деятельности поддерживает до 3 уровней вложенности
- Статическая авторизация через API Key
- Sentry используется для мониторинга ошибок в реальном времени

Все ответы возвращаются в формате **JSON**.

### Основные сущности

**_Organization_** — организация

**_Activity_** — вид деятельности

**_Building_** — здание

**_Phone_** — номер телефона

### Описание связей:

| Связь                          | Тип               | Описание                            |
|--------------------------------|-------------------|-------------------------------------|
| **Building -> Organization:**  | One-to-Many       | Одно здание — много организаций     |
| **Organization -> Phone:**     | One-to-Many       | Одна организация — много телефонов  |
| **Organization <--> Activity** | Many-to-Many      | Через таблицу organization_activity |
| **Activity -> Activity:**      | Self-reference    | Иерархия "родитель-наследник"       |


### Иерархическая структура видов деятельности

- Имеет максимум три уровня
- Уровень определяется положением в иерархии
- Вложенный вид деятельности может иметь одного родителя (ForeignKey)
    
    | Уровень | Тип               |
    | ------- |-------------------|
    | 0       | Автомобили        |
    | 1       | Легковые          |
    | 2       | Аксессуары        |


### Права доступа

- Все запросы для получения организаций требуют статический API-ключ в заголовке

- API-Key: your_api_key

- Настройки API-ключа находятся в файле:

    app/api/dependencies/auth.py

## Мониторинг и обработка ошибок

- Проект оснащён системой мониторинга ошибок **Sentry**, которая позволяет:
  - Автоматически ловить и группировать все исключения в приложении
  - Получать полный стек-трейс ошибки с контекстом запроса (URL, метод, заголовки)
  - Все необработанные исключения, ошибки базы данных и проблемы с запросами автоматически отправляются в Sentry
  - Чтобы использовать Sentry в проекте:
    - Создайте проект на [sentry.io](https://sentry.io)
    - Скопируйте DSN
    - Добавьте в ".env":
      SENTRY_DSN=your_sentry_project_dsn
  - Если вы **не хотите** использовать Sentry — не указывайте переменную "SENTRY_DSN" в ".env". 
  Приложение запустится и будет работать корректно.

- Логирование настроено на уровне сервисного слоя с использованием стандартного модуля "logging"

- Реализованы health-check эндпоинты:
  - "GET /health/" — общий статус API
  - "GET /health/db" — проверка подключения к PostgreSQL + PostGIS

## Запуск проекта

### Через Docker Compose

```
1. Клонировать репозиторий
    git clone https://github.com/simbarilion/GisCatalog
    cd GisCatalog

2. Создать проект на платформе sentry.io (опционально)

3. Создать .env файл и настроить переменные окружения на основе примера (env.example):
    # App
    ENV=development_or_production
    SECRET_API_KEY=come_up_with_your_own_key
    # Database
    DB_NAME=gis_catalog
    DB_USER=postgres
    DB_PASSWORD=your_own_password_here
    DB_HOST=db
    DB_PORT=5432
    # Hosts
    HOST=0.0.0.0
    PORT=8000
    ALLOWED_HOSTS=localhost,127.0.0.1,0.0.0.0
    #Sentry
    SENTRY_DSN=your_Sentry_DSN_Client_Key

4. Добавить GitHub Secrets
    - В репозитории - Settings - Secrets and variables - Actions должны быть добавлены:
        
    | Secret                    | Назначение                               |
    |---------------------------|------------------------------------------|
    | `SECRET_API_KEY`          | SECRET_KEY для статической авторизации   |
    | `DB_PASSWORD`             | Пароль PostgreSQL                        |
    | `ALLOWED_HOSTS`           | localhost,127.0.0.1,0.0.0.0              |

5. Запустить приложение
    docker compose up --build

- После запуска приложение будет доступно по адресу:
    http://localhost:8000

- Документация API:
Swagger UI: http://localhost:8000/docs
ReDoc: http://localhost:8000/redoc
```

### 2. Локальный запуск (без Docker)

```
1. Клонировать репозиторий
    git clone https://github.com/simbarilion/GisCatalog.git
    cd GisCatalog
    
2. Установить зависимости
  - Через Poetry:
    poetry install
  - Через pip:
    pip install -r requirements.txt
    
3. Создать проект на платформе sentry.io (опционально)

4. Создать .env файл и настроить переменные окружения на основе примера (env.example)
    cp .env.example .env

5. Создать базу данных PostgreSQL с расширением PostGIS

6. Примененить миграции:
    alembic upgrade head

7. Заполнить базу данных тестовыми данными (опционально):
    python -m app.db.seed.seed --clear 

8. Запустить приложение
    poetry run python -m app.main

```

## Архитектурные решения

- Многослойная (Layered) архитектура:
    - routes — слой представления (API endpoints)
    - schemas — слой валидации и сериализации (Pydantic)
    - services — бизнес-логика
    - repositories — слой доступа к данным (Repository Pattern)
    - models — слой доменных моделей (SQLAlchemy)

- Разделение ответственности между слоями

- Repository Pattern — отделение бизнес-логики от деталей работы с базой данных

- Иерархия видов деятельности реализована через Self-referential ForeignKey

- Автоматический расчёт уровня вложенности (level) при создании деятельности

- Разделение Pydantic-схем:
  - schemas/request/ — схемы для входящих данных
  - schemas/response/ — схемы для ответов API
  - schemas/common/ — общие базовые схемы

- Геопоиск организаций с использованием PostGIS:
  - поиск в радиусе (ST_MakePoint, ST_DWithin) 
  - поиск в bounding box (ST_MakeEnvelope, ST_Within) 
  - использование геоиндексов (GIST)

- Оптимизация производительности:
  - GIST-индекс на геометрическом поле location
  - B-tree индексы на часто используемых полях (name, building_id)
  - Рекурсивный CTE-запрос в PostgreSQL для получения всех потомков в иерархии видов деятельности

- Маппинг объектов через отдельный слой мапперов (app/mappers/):
  - преобразование SQLAlchemy-моделей в Pydantic-схемы 
  - инкапсуляция логики формирования response-объектов
  - разделение бизнес-логики и сериализации 

- Использование Dependency Injection через систему Depends FastAPI:
  - внедрение зависимости на уровне роутера без повторения в каждом маршруте
  - уменьшение связности между компонентами (модульность)
  - упрощение тестирования

- Поддержка пагинации: limit, offset, общее количество записей (total)

- Использование аннотаций типов

- Централизованная обработка ошибок через кастомные exception handlers

- Логирование настроено с помощью Sentry, а также стандартного logging:
  - все необработанные исключения, ошибки базы данных и проблемы с запросами автоматически отправляются в Sentry 
  с полным стек-трейсом и контекстом запроса
  - структурированное логирование на уровне сервисного слоя

- Health-check эндпоинты для мониторинга

- Прекоммит хуки (pre-commit) для поддержания качества кода

- Контейнеризация приложения с помощью Docker и Docker Compose

- Автоматические миграции базы данных через Alembic

- Конфигурация приложения через переменные окружения (.env)
